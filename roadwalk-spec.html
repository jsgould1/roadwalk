Design document · April 2026

A walk down the road,
turned into data.

A mobile-first field inventory tool for roadway segments and parking lots, designed around the reality of walking a site with a phone in hand. Captures points, lines, polygons, and background tracks; renders a live plan-view ribbon as you walk; delivers tables and plan labels that flow straight into deliverables.

One-sentence charter

Geometry, photo, and type in under ten seconds; everything else deferrable — and the tool adapts to the job, not the other way around.

This document assembles decisions made during design conversation into a coherent picture of the tool. It is deliberately prior to any code. Where questions remain, they are flagged in section 10.

Intent & principles

Why this tool, and why this shape

Past inventory tools built on the same single-file HTML pattern have accumulated capability over time and, in doing so, become slow to use in the field. Forms grew long. Bottom bars filled up. Every feature looked like every other feature. The tool held its value at the desk, but the walk itself got harder.

This tool is a reset on that trajectory. It starts from the walk and works outward.

Principles

Seconds, not minutes
A feature — point, line, or polygon — should be placed in under ten seconds. Photo, geometry, type, one required field. Everything else is deferrable.
Deferral is honest
Features that are partially filled in the field are explicitly marked incomplete. They do not hide. They resurface at the truck, at the desk, at section close. The tool never assumes the walk was complete.
Geometry in the background
Guardrail, fence, curb, swale, and edge drop-off are not stop-and-fill features. They are tracks that run while attention is elsewhere. The screen accommodates several simultaneous tracks without getting in the way.
The screen is a plan sheet
The ribbons running down the left and right edges of the screen are a live, schematic plan view. Drafting conventions — solid curb, dashed guardrail, double curb-and-gutter, >>> swales — render as you walk. The field display and the deliverable speak the same visual language.
Configurable, not hardcoded
Feature types, codes, waypoint forms, kickoff and closeout questions, and ribbon symbology live in settings. Different jobs need different tools. The app adapts to the job.
Progressive disclosure
Forms start minimal and grow only where upstream answers demand it. An ADA ramp that is compliant does not unfold PROWAG deficiency fields. A grate inlet shows grate condition; a curb inlet does not.
Offline by default
Everything works with no connection. Data lives in IndexedDB. Sync is a later step, not a prerequisite.

Data model

Three nested containers, and the things they hold

The tool organizes around three nested levels: the project (the job), the section (a single road segment, parking lot, apron, etc.), and the feature (a sign, a culvert, a repair, a length of guardrail). Each level carries its own metadata; each lower level inherits context from the one above.

Data Model Overview Diagram 2.1

What travels with each level

A project holds the catalog that its sections draw from — a subset of the global feature-type master, optionally augmented with project-specific one-offs. It also holds the alignment library, the crew roster, and the site boundary. One project, many sections.

A section is the unit of a walk. It has a type (linear or area), a boundary, a set of kickoff answers that configure its UI, and — at the end — a set of closeout recommendations that flow to the deliverable. A crew works one section at a time, but switches between sections freely via the header dropdown.

A feature is a physical thing inventoried within a section. It inherits context from its section — if the section is linear, the feature carries station values; if area, it carries GPS extents only. Features follow a progressive-disclosure form tree defined in the catalog.

Section types

Two modes, shared point features, different linear treatment

Every section is either linear or area. This distinction drives a great deal — whether the ribbon is active, whether stationing applies, how repairs are authored, how certain features (edge drop-off in particular) are captured. Point features — signs, catch basins, monuments, ADA ramps — behave identically across both modes.

Both types also carry a required pavement type attribute set at kickoff: asphalt (AC), Portland cement concrete (PCC), composite (AC over PCC), chip seal, gravel / unpaved, or other. Pavement type drives which distress vocabulary is offered (joint reflection cracking is a PCC-on-AC concept; faulting is a PCC concept; raveling is an AC concept), which photo subjects make sense, and how repairs are described. Mixing types within one section is allowed but flagged at closeout — the typical resolution is to split the section.

Either type of section can be split into sub-sections at any point — useful for long roadways that report better as multiple shorter runs, or for one-polygon parking sites that contain three functionally distinct lots. Splits use a name::01 / ::02 naming convention and reassign features by station (linear) or point-in-polygon (area). See § 9 for the mechanic.

Section type · Linear

Roadway segment

Alignment-based. Stationing is the native coordinate. The ribbon is active, showing plan-view symbology on both sides of the pavement cut line. Repairs can be parametric (lane + position + length) or drawn. Background tracks — guardrail, fence, curb, swale, edge drop-off — run while walking.

Used for: road segments, taxiways, runway shoulders, rail intermodal access roads, any linear facility where centerline is meaningful.

Section type · Area

Parking lot, apron, pad

Polygon-bounded. No stationing, no ribbon. Features collected within the polygon use GPS extents only. Edge conditions become polylines along the perimeter or polygons; repairs are always drawn. Kickoff asks stall counts, ADA stalls, fire lane, striping, drainage, curb stops (count + general condition), and the early distress impression by category.

Used for: parking lots, truck aprons, container yards, helipads, any facility where relative position matters more than station.

Feature catalog

What the tool knows how to collect, and how each thing behaves

The catalog is a settings-editable master list. Each entry defines a feature type's name, two-letter plan-label code, geometry, and — for track-type features — ribbon symbology. Entries also carry a form tree describing what questions to ask and when.

The catalog is hybrid: a global master list is maintained across projects; each project selects its active subset and may add project-specific types. Inactive types stay in the master but drop out of the field UI so the bottom bar stays short.

Lifecycle of a walk

From parking the truck to delivering the report

A walk has a shape. Kickoff configures the walk; collection fills it; closeout validates it; export hands it off. Each stage has its own screens and its own rhythm. The tool pushes crews through that shape without getting in the way.

Walk Lifecycle — Seven Stages Diagram 5.1

How it feels in sequence

01 · Open project. App launches to the project list. Crew picks the job, lands on the section list. Each section shows a status pill: not started, in progress, or closed.

02 · Kickoff. Tapping an unstarted section opens the kickoff interview. Hard-gate questions (direction of travel, lane count, is this curbed, distress impression by category — load-related, climate-related, other) must be answered because they drive UI behavior and prime the inspector's eye for the walk ahead. Soft questions (estimated PCI numerical value, refined per-distress severities, lot details like exact curb stop count) can be deferred to closeout. The early distress-impression question takes seconds — three category toggles with a short multi-select chip group under each — but it sets the mental frame for what the crew is looking for and feeds the closeout reconciliation later.

03 · Alignment. For linear sections, the alignment is either imported (GeoJSON, KML, KMZ) or drawn in the app on top of the selected basemap. Station 0+00 is set and direction locked. Area sections skip this step — they draw a polygon boundary instead.

04 · Collect. The field view. This is where the crew spends 95% of their time. Map dominates the screen; top bar holds active track chips; ribbons run down both edges; bottom bar holds the "add feature" and "start track" actions; tally bar is persistent. Most interactions complete in under ten seconds.

05 · Review. Back at the truck, or that evening, crew opens the section's feature list and works through anything flagged incomplete. Fields that were deferred during the walk get filled now, while memory is still fresh.

06 · Closeout. When the section is complete, the closeout screen pulls auto-tallied data (repair count, total crack seal LF, drainage issues flagged), surfaces reconciliation items (kickoff said curbing but no curb track collected), and asks for recommendations (needs overlay, needs geotech, ADA concerns, safety concerns).

07 · Export. At project level, deliverables roll up. Inventory tables per section. Plan-label KML for drafting. Photo archive keyed to feature IDs. Executive summary PDF. Whatever the job demands.

The ribbon, in detail

Live plan-symbology strip, one per side of the road

The ribbon is the tool's signature element. It is a narrow vertical strip along each screen edge that renders a live, schematic plan view of the road cross-section at the current station — and the 100 ft or so behind it. As the crew walks and tracks fill in, the ribbon fills in with them. What appears on the ribbon is the same drafting language that will appear on the plan sheet.

Slot structure

Each ribbon carries five fixed slots, ordered outside-in from the screen edge toward the pavement. The slots are always there; what renders in them depends on which tracks are active on that side. Pavement lanes occupy additional thin slots inside the cut line, but those are driven by lane count from kickoff, not by tracks.

Ribbon — Annotated Cross-Section Diagram 7.1

Heat encoding for edge drop-off

Edge drop is the only track whose ribbon output carries two independent scalars: depth and width. Depth drives color; width drives stroke thickness. Between waypoints, the displayed value is the most recent measurement — the ribbon updates in steps, not in gradients.

SwatchDepth rangeSemantics
Measured zero Edge drop confirmed absent at waypoint
0 – 2 in Acceptable; noted for record
2 – 4 in Marginal; candidate for shoulder rebuild
4 – 6 in Deficient; flagged on deliverable
> 6 in Hazard; triggers safety recommendation

Width caps the heat band at about 5 ft equivalent on screen. Drops wider than that render with an <> overflow indicator rather than growing further — the screen is narrow, and visual accuracy matters less than the fact that the drop is wide.

Pacing indicator

A small chip at the top of each ribbon shows distance since the last waypoint on that side's most-recently-touched track. Green under 50 ft, yellow 50–100, red over 100. This exists because the single biggest failure mode of track features is forgetting to take a waypoint for a long stretch — the map and the chip row don't make that obvious, but a pacing chip can.

Body-relative orientation

The ribbon is always oriented walk-forward-is-up, independent of map rotation. A crew walking south on a northbound road sees north-up on the map and still sees the current station advancing up the ribbon as they walk. Left chip row is their physical left; right chip row is their physical right. This is the piece that most complicates the rendering math, but the alternative — map-orientation-dependent ribbon — is disorienting in the field.

Tally bar, in detail

Crack-seal totals that three capture paths roll up into

Crack seal is a quantity-driven line item. Mile after mile of incidental cracks at unknown locations do not deserve a feature each; they deserve a running total. The tally bar is where that total lives. It stays visible on the field screen so the number is never ambiguous.

Tally Bar — Annotated Diagram 8.1
current GPS of the taper — not B's location. Recent strip prepends a ~ (tilde) on B entries to mark stationing as approximate.

Where the running total comes from

Three paths sum to one number:

1 · Increments
The +10 / +50 / +N buttons in the tally bar, attributed to A or B. Partner (B) values are stationed with a tilde to mark them approximate.
2 · Drawn CS
Any crack seal polyline feature captured on the map as a discrete patch contributes its length to the tally. These are feature-level and carry photos.
3 · Closeout adjustment
At closeout, the crew can override the total with a note — useful when partner estimates drift or when a summary number is provided by a contractor at end of day.

Area-section behavior

In a parking lot section, there is no station. Increments time-stamp instead of station-stamp, and the recent strip shows HH:MM. The math and buttons are otherwise identical.

Architecture notes

How the pieces hold together under the surface

The tool follows the single-file HTML pattern established in prior builds: one HTML file, no bundler, no runtime dependencies except the browser. Storage is IndexedDB. All rendering is vanilla DOM and SVG. Map rendering uses Leaflet loaded from a local copy bundled with the file — no CDN dependencies at runtime.

Storage model

One IndexedDB database per installation. Object stores:

projects
Project records. Key: UUID. Value: name, client, catalog subset, alignment refs, crew, site boundary.
sections
Section records. Key: UUID. Value: project FK, type (linear/area), boundary geometry, kickoff answers, closeout recommendations, status.
features
All collected features. Key: UUID. Value: section FK, type code, geometry (GeoJSON), station range (linear only), attributes, photos (blob refs), device_id, client_created_at, client_updated_at, completeness flag. The device and timestamp fields exist from v1 to keep v2 sync additive rather than a rewrite.
waypoints
Track waypoints. Key: UUID. Value: feature FK (of the track), position (GPS + station), attributes (e.g. depth/width for ED), photo ref, timestamp, attribution (A or B).
tally_entries
Crack-seal tally log. Key: UUID. Value: section FK, amount, attribution, GPS, station, approximate-flag, timestamp.
daily_reports
One record per project per crew-day. Key: project FK + ISO date. Value: crew roster (with A/B device mapping), weather, surface state, time-on-site per inspector, sections worked, equipment used, free-text journal, pre-walk checklist responses. The day's Sa features are joined in at render time, not duplicated here.
sketches
Standalone sketch records. Key: UUID. Value: parent (project / section / feature) FK, backdrop type (blank / aerial-image / section-polygon / alignment), vector annotation JSON, title block metadata, photo blob ref for the rendered output, scale factor for measurement.
catalog
Master and per-project feature type definitions.
photos
Per-photo metadata records. Key: UUID. Value: parent feature/waypoint FK, blob refs (original + thumbnail), annotation layer (vector JSON), watermark override, capture context (GPS, station, timestamp, device_id, photographer A/B), derived_cache ref to the most recent rendered watermarked-and-annotated version. The photos store sits between feature records and the raw blob bytes, owning all the metadata and overlays so the original blob never needs to change.
blobs
Image binaries. Each record carries a kind field (original, thumbnail, or derived) and a parent photo UUID. Originals are immutable after capture; thumbnails are immutable after generation; derived (watermarked + annotated) blobs are a lazy cache, regenerated on demand and evicted under storage pressure.
snapshots
Rolling JSON backups of the active project. Key: timestamp. Value: full records bundle (no blobs). Retention is capped to the last N snapshots (default 10). See Data durability below.

Section splits

A section that started life as a single unit can be subdivided in the field or at the desk. Two motivating cases recur: a long roadway that turns out to be more useful broken into shorter inspectable runs, and a parking site that reads as one big lot but functionally contains several distinct lots that should report separately.

Naming convention. Splits append a zero-padded ordinal suffix using a double-colon separator. Splitting "Lot A" produces "Lot A::01" and "Lot A::02". Splitting "Main St 0+00 to 50+00" produces "Main St 0+00 to 50+00::01" and "::02" (with their own station ranges). The suffix is part of the display name, not a separate field; the section's UUID stays stable so feature references never break. Re-splitting an already-split child appends another suffix — "Lot A::02::01" — which is visually clunky but unambiguous and rare in practice.

Linear-section splits. The crew taps Split section, then either enters a station value (e.g. "split at 25+00") or drags a divider on the alignment overview map. Features and waypoints with station less than the divider stay with ::01; those greater move to ::02; ties default to ::01. The alignment polyline is shared by reference between the two children — splits do not duplicate or fragment the underlying alignment geometry. Each child gets its own station range, kickoff answers, tally, and closeout. Tally entries are reattributed by their stamped station; entries with no station (rare) are flagged for manual reassignment.

Area-section splits. The crew taps Split section, then draws a dividing polyline through the section's boundary polygon — either a single cut producing two children, or multiple cuts producing N children. Point features are reassigned by point-in-polygon. Linear and polygon features that cross a cut are flagged for the crew to decide: assign whole to one side, or duplicate-and-clip into both. The default is "ask," because guessing wrong here loses information that's expensive to re-collect.

What carries over to the children. Kickoff answers are copied to both children with a "review per section" flag, so the crew is prompted at the next open of each child to confirm or modify. The watermark template, photo settings, and catalog subset are project-level and unaffected. Closeout is per-child and starts fresh. The original section's UUID becomes ::01's UUID; ::02 onward get new UUIDs. This means historical references to the un-split section silently resolve to ::01, which is the conservative behavior.

Reverse operation. Two sibling children can be merged back via Merge with sibling. Features combine, station ranges concatenate (linear), polygons union (area), tallies sum, and the merged section takes the parent's name without a suffix. Kickoff answers from both children are preserved as a diff in the merged section's notes for the crew to reconcile. Merge is more dangerous than split because it can erase distinctions the crew cared about; it requires a confirmation dialog that lists what will be lost.

Alignment projection

For linear sections, every GPS fix must be converted to a station value along the alignment. The projection is a perpendicular-drop from the GPS point to the alignment polyline, returning both the station (distance along the polyline from 0+00) and the perpendicular offset (signed — left or right).

Two practical concerns. First, the crew does not walk a straight line — they zigzag, step off the road, come back. A naïve projection will assign station values that oscillate. The tool uses a monotonic-constrained projection that only advances or stays put, never retreats, as long as the crew is still moving forward in the section. Direction reversals are explicit (a toggle in the header), not inferred.

Second, offset direction (left vs right) is resolved against the alignment's tangent and the recorded direction of travel. "Left" is always the crew's physical left while walking the alignment in the kickoff-declared direction — this is what drives chip rows and ribbon sides consistently.

Offline-first behavior

Everything works with no connection. Basemap tiles are cached opportunistically when online (section kickoff is a natural time to pre-cache). All data writes go to IndexedDB synchronously. In v1, a project is exported as a bundle (JSON + photo archive) and re-imported on the office machine. Multi-device sync is a v2 feature, described below — but the data model is built to support it from the start.

Multi-device collaboration v2

Crews routinely work a site as a pair — one walking each side of the road, or splitting a lot between them. In v1, each phone captures its own section and the two bundles are merged at the desk. In v2, the two devices pair in the field and sync their work in real time.

The v2 architecture uses a QR-code handshake plus WebRTC DataChannel, entirely peer-to-peer, no server required:

  • Pairing. Phone A generates a QR code containing a WebRTC offer. Phone B scans it with the in-app camera, generates an answer, and displays its own QR. Phone A scans that. Total handshake: ~15 seconds. Animated QR is used if the payload exceeds the capacity of a single static code.
  • Transport. Once paired, the two devices hold a bidirectional DataChannel. Features, waypoints, and tally increments stream between them as small JSON messages. Works on iPhone and cellular iPad equally — they are functionally the same in Safari for this purpose.
  • Network. The expected field setup is one device running Personal Hotspot, the other(s) joined to it. Pure local-network peer-to-peer; works without cellular signal, since the hotspot creates a local WiFi regardless. Cross-cellular pairing (each device on its own carrier) is a best-effort attempt via a public STUN server, with graceful fallback to "get on the same network."
  • Merge model. Features and waypoints are CRDT-friendly: each record carries a UUID, a source device_id, and a created_at timestamp. Concurrent creation on two devices is never a conflict — both records are kept. Edits on the same record are resolved last-write-wins at the field level, with the loser's value preserved in a revision history for recovery. The tally bar treats increments as additive-only (never overwritten), which sidesteps conflict entirely.
  • Attribution carry-over. The A/B labels chosen at section kickoff map to the paired devices. When B taps +50 on their phone, A's phone shows the total and the recent strip updates with B's contribution on B's row.
  • Drop handling. Connection loss is expected and non-fatal. Each device continues capturing locally; on reconnection, a delta sync catches the other up. If the devices never reconnect in the field, the bundles still merge cleanly at the desk. A persistent sync-status chip in the field-view header shows the current state at a glance — Synced (green, all records acknowledged by peer), N pending (yellow, items captured locally but not yet confirmed across), or Disconnected (red, peer unreachable). Tapping the chip opens a small panel listing unsynced records and offering a manual retry. The chip also displays in solo mode as a neutral "Solo" pill, so the crew always knows what they are in.

Two data-model decisions are made in v1 to avoid retrofit pain later: every record written to IndexedDB carries device_id and client_created_at from day one, even in single-device mode. Sync is additive when we get to it, not a rework.

Dimensional capture / LiDAR v2+

iPad Pro (2020+) and iPhone Pro (12 Pro+) ship with LiDAR sensors accurate to roughly centimeter-level out to ~5 m. Fused with the camera via ARKit, they enable in-field measurement of sign face dimensions, guardrail heights, edge drop-off depth/width, repair polygon outlines (walk the perimeter and get an accurate shape in seconds), and post spacing on linear features. For several feature types, LiDAR would replace a ruler or an eyeball estimate with a measurement an inspector can defend.

The blocker for the v1 architecture: iOS Safari does not expose LiDAR or ARKit to web pages. A single-file HTML tool cannot reach the depth sensor. "Measure from an existing photo" is also a misconception — LiDAR records depth at capture time alongside the camera frame; a plain photo taken later cannot be retroactively measured.

Three paths to unlock LiDAR, in ascending scope:

  1. URL-scheme handoff to a capture app. On LiDAR-eligible feature types (sign, guardrail, repair polygon), a "LiDAR capture" button launches Polycam or Canvas via iOS URL scheme. The inspector scans. The resulting model (OBJ/USDZ) or measurement (length, area) is imported back into the feature record as an attachment or an attribute. Zero native code; relies on the capture app being installed. Lowest scope, real value for repair polygons specifically.
  2. Capacitor wrapper with an ARKit plugin. Wrap the existing HTML in Capacitor. Write a small Swift plugin exposing LiDAR-backed measurement primitives (point-to-point distance, rectangle-on-plane, polygon-on-floor) to JS. Preserves the web codebase; adds an Apple Developer account, signing, and distribution mechanics. Unlocks in-app measurement without a full rewrite.
  3. Native iOS app. Full rewrite in SwiftUI with native ARKit integration. Highest capability, lowest leverage from what we've built. Not recommended unless a client deliverable explicitly requires it.

Where LiDAR earns its keep. Best ROI: repair polygons (walking perimeter is the slowest common interaction, a scan is seconds). Moderate ROI: edge drop-off (only when depths actually vary; "copy last" is already fast for uniform sections). Low ROI: sign sizing (MUTCD codes imply size) and guardrail height (categorical compliance is usually the deliverable, not exact height). Two LiDAR-capable iPads do not multiply the LiDAR benefit — one scan per feature is enough — but they compound with the v2 WebRTC split-walk pattern for partner-pace data entry.

Recommendation. v1 ships without LiDAR. Path 1 (URL-scheme handoff) is the right bolt-on for v2 if any client engagement prioritizes dimensional accuracy. Path 2 is a v3+ consideration if LiDAR becomes central.

Data durability & auto-backup v1

IndexedDB is durable across app restarts and phone restarts — every capture is persisted the moment it's written. The risk the tool defends against is not transient memory loss but device-level loss: a phone cracks, drowns, is stolen, is left on the tailgate and driven over, or its browser storage gets evicted under iOS storage pressure. Against any of those, the IndexedDB copy is gone with the device.

The tool therefore runs a rolling auto-backup on a settable interval, writing a full JSON snapshot of the active project to the device's Files app (iOS) or Downloads folder (Android). The default interval is 15 minutes during an active walk; settings allow 5, 10, 15, 30, or 60-minute options, plus on section close and on app background triggers. Backups are cheap — a typical working project is a few hundred KB of JSON, plus photos referenced by UUID. Photos are not duplicated into each backup; they remain in the IndexedDB blob store and are also mirror-cached to a device-level folder on capture.

Two technical constraints shape the design:

  • iOS Safari cannot silently write to the file system. Every save requires a user gesture. The auto-backup therefore runs as a prompt, not a true background job: at each interval, a non-blocking banner appears at the top of the screen — "Backup ready · tap to save" — and remains until tapped, dismissed, or the next interval arrives. Tapping writes the JSON snapshot via the standard download anchor, landing in the Files app. The prompt is designed to be a single confident tap: no file picker, no naming dialog, just write.
  • Section close is a hard gate. A section cannot be marked closed until its most recent backup is confirmed saved within the last 5 minutes. This is the one case where the backup prompt is blocking rather than dismissable, because closing a section is the point at which the crew mentally moves on from that data.

In addition to the file-system backup, the tool maintains an in-database snapshot store — a second IndexedDB object store holding the last N full-project snapshots (default: last 10). This is silent, cheap, requires no user gesture, and provides a rollback path if a catalog edit or bulk operation corrupts the working set. It does not defend against device loss, but it defends against the crew's own mistakes.

Settings surface three related controls: backup interval, number of in-DB snapshots to retain, and a one-tap Export backup now action for when the crew wants an explicit save before, say, walking into a building with unreliable GPS or handing the phone to someone else.

Phone-loss survival. The Files-app backup is the only layer that protects against physical destruction of the phone, and only when iCloud Drive is enabled for the Files app. With iCloud on, every backup is mirrored to iCloud within seconds and recoverable from a replacement device. Without iCloud, the backup lives only on the phone — same risk surface as the IndexedDB it copied from. The setup checklist for a new field device should include a confirmation that iCloud Drive is on for Files and that the user has authenticated.

Photo handling

Photos are first-class artifacts of the inspection record, not just attachments. The pipeline supports capture, on-device processing, in-app markup, and multi-version export — with the original always preserved, untouched, as the source of truth.

At capture. The camera API hands back a JPEG. The tool immediately produces three artifacts and writes them to IndexedDB:

Original
Source image, downscaled to a max long-edge of 1920 px (configurable per project; some clients want full-resolution originals retained). Target ~200–400 KB. EXIF is stripped except for a synthetic timestamp and GPS block written from the device's current fix. This file is never modified after capture.
Thumbnail
256 px long-edge JPEG generated via canvas in the same capture handler. Used for fast scrolling in the in-app gallery, the deliverable PDF, and any list view. Generating thumbnails at capture rather than on demand avoids stutter when the gallery is opened over IndexedDB on a slow device.
Annotation layer
Empty at capture. A JSON document of vector annotations (strokes, arrows, circles, text callouts) layered over the original at render time. Stored in the photos record, not baked into the image. This is what makes annotations editable, removable, and re-stylable later.

Watermark template. A project-level setting defines the watermark applied at render and export time (not at capture). Default template is a bottom-left corner stamp: a compact semi-transparent black block with white text containing project name, section, station (linear sections), GPS, capture timestamp, and a small AECOM logo above the metadata. Sized to occupy roughly the lower-left quarter at most, leaving the rest of the frame clean for the subject. Per-photo overrides exist (e.g. add a "DRAFT" diagonal stamp, swap to a partner's logo for a co-stamped project, or suppress the watermark entirely for one shot). Like annotations, watermarks are never baked into the stored image — they are composited at export.

In-app markup. Tapping a photo opens a full-screen viewer with annotation tools at the bottom: pen, arrow, circle/rectangle, text callout, color picker (5 fixed colors tuned for visibility on pavement: red, yellow, lime, white, black), undo, clear-all, save. Touch input is captured to canvas; strokes are stored as vector data in the annotation layer JSON, not rasterized. This means a small annotation file (a few KB) accompanies each photo regardless of what was drawn — and re-rendering at any size stays crisp.

Storage model addendum. The blobs store now holds two records per photo (original, thumbnail), each with a kind field. A new photos store sits between feature/waypoint records and the blob store, owning per-photo metadata: blob refs, annotation layer, watermark override, capture context (GPS, station, timestamp, device, photographer), and a derived_cache of the most recent rendered watermarked-and-annotated version (rebuilt lazily, evicted under storage pressure).

Photo export surface

The project export expands to four photo variants, each opt-in at export time. The crew picks which subset to include based on what the deliverable is for — internal QC, client report, archival, etc.

originals/
The unmodified capture (resized only by the at-capture downscale). No watermark, no annotations. Use case: handing raw evidence to legal, or re-processing externally with different watermark templates.
watermarked/
Resized (default 1200 px long edge, configurable) with the watermark band composited in. No annotations. Use case: client deliverables where the photo speaks for itself but the metadata needs to be on-image for chain-of-custody.
annotated/
Resized with both watermark and annotations composited in. Use case: condition assessment reports, distress documentation, work-order back-up. This is the typical "report photo."
thumbnails/
256 px versions, no watermark, no annotations. Use case: report indexes, inline gallery in PDFs, or anywhere a fast-loading preview is needed without burdening the page count.

Photos are renamed at export to {section}_{code}_{featureId}_{idx}.jpg within their respective subfolder, so the same logical photo carries the same name across all variants. A manifest photos.csv at the archive root maps each filename to its parent feature, capture metadata, and which variants are present, making downstream batch processing and report generation straightforward.

Export surface

One project-level export action produces:

  • Inventory tables (CSV, one per feature type, keyed by two-letter code)
  • Plan-label KML (one layer per feature type, labels carry code + feature ID)
  • Executive PDF (auto-tally summary + reconciliation status + recommendations per section, with thumbnail strip per section)
  • Photo archive (ZIP, organized into the four variant subfolders described above; crew selects which variants to include)
  • Raw bundle (JSON of all records + originals + annotation JSON, for re-import — re-import preserves edit history)

Export runs entirely client-side — no server round-trip. On iOS Safari, large ZIPs download via File System Access API where supported, otherwise via object-URL anchor download.

Daily collection report

Each calendar day a crew is in the field, the tool maintains a daily report — a project-level artifact, not section-level, capturing the conditions and people behind the day's data. This is the document QC needs when a deliverable is questioned six months later, and the document the office needs to bill the day correctly.

Fields are minimal and entered once at the start and end of the day:

Date
Auto, from device clock at first capture of the day.
Crew
Names of inspectors on site, mapped to A/B device attribution. Free text per slot. The crew list is suggested from prior days for fast pick.
Weather
Temperature, conditions (sunny / overcast / rain / snow / wind), pavement surface state (dry / damp / wet / icy). Manual entry; an optional weather API auto-fill is a v2 nicety. Captured at start of day; updateable mid-day if conditions change materially.
Time on site
Start and end timestamps per crew member. Used for billing and for deliverable QC ("PCI was rated by inspector X, who was on site from 0900 to 1530").
Sections worked
Auto-derived from any section with captures dated this day. Manual override for "we walked it but didn't capture anything."
Safety log
Auto-populated from the day's Sa (safety observation) features, plus a free-text block for end-of-day debrief notes that don't merit a pin (PPE issues, near-miss, things to flag for the next day's crew).
Equipment / vehicle
Truck or vehicle ID, any equipment used (drone, retro-meter, GPR, hand odometer). Useful for QC trace-back.
Notes
Free-text journal for the day. The "what happened that the data alone won't tell you" entry.

Daily reports are stored in their own daily_reports object store (project FK + date as composite key, one record per crew-day) and roll up into the project export as a daily_log.csv plus a per-day PDF page in the executive summary.

Safety module

The safety module is two things working together: the Sa point feature for capturing hazards and observations in the moment during a walk, and the safety log on the daily report that aggregates them. The intent is low-friction capture — when an inspector sees something unsafe, they drop a Sa pin in two taps, the way a note feature is dropped, and the rest of the day's report fills itself.

Optional pre-walk safety checklist (settings-configurable per project): PPE on, two-way comms checked, traffic control plan verified, hazard awareness brief given. Captured once per day at first launch, stamped to the daily report. Not a hard gate — bypassable with a tap — because the tool's job is to record, not to enforce.

For projects with formal safety reporting requirements (some clients require Job Hazard Analysis filed daily), the daily report PDF includes the safety log as a section that can be detached or printed standalone.

Sketch tool

Independent of photo annotation, the sketch tool produces standalone sketches: a blank canvas, an aerial-image background, or a section's polygon/alignment as backdrop. Same drawing primitives as the photo annotation tool (pen, arrow, shapes, text, colors, undo, clear), with an added "measure" tool that draws a labeled distance line based on the underlying scale.

Sketches attach to a project, a section, or a feature. They export to the deliverable in two forms: as PNGs in a sketches/ archive subfolder, and as PDF pages in the executive summary, sized to print clean on letter-size paper. Each sketch carries a title block with project, section, date, and inspector — composited at export, same model as photo watermarks.

Use cases in the field: locating a sample unit for re-find ("here, 25 ft east of the third light pole"), sketching a complex distress that doesn't photograph well, drawing a recommended traffic-control diagram for a repair, capturing a partner's verbal description of a feature too far to walk to.

Desktop summary mode

The same single-file HTML serves a desktop layout for back-at-the-office work. The viewport detects width and switches modes — under ~900 px presents the mobile field UI, above presents a multi-pane summary layout with the project's data laid out as filterable tables, with the map alongside. Nothing the desktop mode does is unavailable on mobile; it's a layout optimization, not a separate app.

Desktop mode focuses on three workflows:

  • Review and complete — work through flagged-incomplete features, fill deferred kickoff fields, run closeout reconciliation across all sections at once.
  • Tabular reporting — every feature type as a sortable, filterable table with all attributes visible. Copy-out to spreadsheet or paste into reports. The same data the export CSVs contain, browsable in-app.
  • Print preparation — page-broken layouts of the executive summary, daily reports, and sketches, with print CSS that produces clean letter-size output. The "print final report" workflow that doesn't require leaving the tool.

Architecturally this is just additional CSS media queries and a layout component — no new data, no new storage. The mobile field UI is the source of truth; desktop mode is a different lens on the same records.

Basemap sources

Licensing constrains basemap choices. Google and NearMap require API keys (AECOM may have or be able to provision enterprise keys); OSM and Esri World Imagery are free to use within their terms. Street View requires a Google key; Mapillary is the free alternative and may be adequate for inspection confirmation. The basemap source is a settings-level choice, swappable per project.

Open questions

Decisions still to make before build

These surfaced during design and were deferred. They are listed here so they do not disappear into the scaffolding. Each is tagged for priority: low can wait until v2; v1 is a blocker for first build; unpriortized items need a judgment call.

Naming & conventions

v1
Code casing. Lock one convention — all title case (Si, Cu, Gr), all upper (SI, CU, GR), or hybrid. Current working list is mixed. Recommendation: title case for readability, with catalog editor enforcing uniqueness.
low
Section naming. Free text or controlled from a template? Free text is faster to start; controlled names make exports cleaner.

Ribbon extensibility

v1
Lane count beyond 2 per side. Ribbon currently accommodates shoulder + L1 + L2. Roads with 3–4 lanes per side (highway mainline, some urban arterials) need a strategy — wider ribbon, collapsed inner lanes, or a tap-to-expand model.
low
Opposing-direction traffic. When a two-way road is walked in one direction, does the opposite direction's lane count render on the far-side ribbon? Probably yes for completeness, but adds complexity.
Shoulder width variability. Single value per side, per direction, or stationed. Probably single per side with a shoulder changes mid-section escape in kickoff.

Repairs

Repairs spanning NB+SB lanes. One polygon crossing the centerline, or two distinct polygons? Leaning one polygon with a crosses centerline flag for the deliverable.
low
Lane-exact geometry. Parametric repairs currently assume centerline + lane-offset geometry. Lane-edge snapping from imported geometry would be more accurate but requires richer alignment data.

Field ergonomics

Swale vertex capture. Auto-simplify from GPS trail (fewer taps, less control) vs tap-to-add vertices (more control, more effort). Probably hybrid: auto-capture with tap-to-override.
Polygon perimeter capture. Walk-the-perimeter with live trail or tap-the-corners at a distance. Parking lots especially — may need both modes.
low
Section switching mid-feature. If the crew opens a waypoint sheet and then switches section, what happens to the unsaved waypoint? Likely a modal prompt to save, discard, or stay.
Depth interpolation. Edge drop heat between waypoints: stepped (show last measured value until next) or smooth (linear interp). Leaning stepped — matches reality; doesn't imply precision we don't have.

Data & export

v1
Partner-reported station stamping. Document convention: taper reports ~30 ft seal, tally bar stamps that with taper's current station as approximate. Enshrine the tilde convention and surface it in the deliverable.

Phased build

What ships first, what waits

The tool is ambitious. Building all of it in one push invites the same accretion problem prior tools have suffered. The phased breakdown below separates the bones of the walk — the things without which the tool is pointless — from the refinements and edge cases.

Version 1 · The walk works

The minimum viable walk. A linear section can be kicked off, walked, collected, closed, and exported. Everything else is deferred until this is proven in the field.

  • v1 Project → section → feature data model and IndexedDB storage
  • v1 Linear and area section types
  • v1 Section splits and merges — name::01 / ::02 naming, station-divider for linear, draw-cut for area, point-in-polygon feature reassignment, merge-with-sibling reverse op
  • v1 Kickoff interview with hard-gate / soft-defer split
  • v1 Alignment draw-in-app (polyline tap) and GeoJSON/KML import
  • v1 Perpendicular GPS → station projection with monotonic constraint
  • v1 Full catalog of point features (Si, No, CB, In, MH, CO, GG, Mo, AR, PS)
  • v1 Linear features: Cu, Ga, CS
  • v1 Track features with chip bar, waypoints, and ribbon symbology: GR, Cb, Sw, ED
  • v1 Parametric repairs and drawn repairs
  • v1 Tally bar with three capture paths, A/B attribution, recent strip
  • v1 Ribbon rendering: 5-slot cross-section, heat encoding, station ticks, pacing indicator
  • v1 Waypoint sheet with copy-last and recent strip
  • v1 Section closeout: auto-tally, reconciliation, recommendations
  • v1 Catalog editor (JSON-backed, minimal UI)
  • v1 Project export (CSV, KML, PDF summary, photo ZIP, raw bundle)
  • v1 Photo pipeline — capture with auto-thumbnail and EXIF strip, project-level watermark template (bottom-band metadata + corner logo), photos store with per-photo metadata, four export variants (originals, watermarked, annotated, thumbnails)
  • v1 Photo annotation tool — full-screen viewer with pen, arrow, circle/rect, text callout, color picker, undo/clear; vector storage so re-render stays crisp at any export size
  • v1 Rolling auto-backup — JSON snapshot to device Files at settable interval (default 15 min), forced before section close, plus rolling in-DB snapshot store for rollback
  • v1 Sync-status chip in field-view header (shows "Solo" in single-device mode; ready for v2 sync states without UI rework)
  • v1 Split-walk operational note — when two phones cover one project in v1 (no sync yet), each crew member exports their bundle to Files at section close; bundles merge at the desk on import
  • v1 Pavement type as required kickoff field (AC, PCC, composite, chip seal, gravel, other) driving distress vocabulary and repair description
  • v1 MUTCD sign code picker for Si feature — searchable, grouped by series, with shape preview; free-text fallback for non-standard signs

  • v1 Desktop summary mode — same single-file HTML, CSS-driven layout switch above ~900 px viewport; tabular reporting, bulk review/complete, print preparation
  • v1 OSM and Esri basemaps

Refinements based on what the first walks teach. Expect this list to change as field experience accumulates.

  • v2 Fence (Fe), Swale (Sl), traffic loop (TL), marking polylines (PL, PT) and marking polygon (PA)
  • v2 3+ lanes per side on the ribbon
  • v2 Opposing-direction lane rendering on the far-side ribbon
